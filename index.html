What is a Java GC Log Parser?

A Java GC log parser is a tool that reads garbage collection logs produced by the JVM and extracts meaningful metrics from them. When you enable GC logging with flags like -Xlog:gc*:file=gc.log, the JVM writes detailed information about each garbage collection event. A parser transforms this raw text into structured data: pause times, heap sizes, GC frequency, and more.

This tool parses your GC logs and visualizes the data, making it easy to spot performance issues without manually reading thousands of log lines.

How to Use This Java GC Log Parser

  1. Generate a GC log: Run your Java application with -Xlog:gc*:file=gc.log:time
  2. Wait for data: Let your app run under production load for 5-10 minutes to collect enough GC events
  3. Paste log: Copy the entire gc.log content and paste it into the textarea above
  4. Parse: Click "Parse Log" to analyze the data

Understanding Java GC Log Parser Results

Pause Time (ms): How long the application was paused during garbage collection. Lower is better. Target varies: 1ms for real-time systems (use ZGC), 50-200ms for web services (use G1GC), 500ms+ acceptable for batch jobs (use Parallel GC).

Heap Before/After: Memory occupied before and after garbage collection. If "After" is consistently high and keeps growing, you likely have a memory leak. If "Before" is very large relative to your heap limit, your heap is too small.

GC Type: The kind of collection (Young, Mixed, Full). Frequent Full GCs are a red flag — it means the old generation is overflowing. Young GCs are normal and expected.

Event Count: Total number of GC events during the log period. If you see 100+ events in 5 minutes, your garbage collector is working too hard — consider increasing heap size or optimizing object allocation.

Common Patterns to Look For in GC Logs

Rapidly Growing Heap: If "Heap After" increases steadily throughout the log (e.g., 100MB → 200MB → 400MB), you likely have a memory leak. Use Eclipse MAT or JProfiler to find which objects are accumulating.

Frequent Minor GCs (Young Generation): If you see 10+ Young GCs per minute, your Eden space is too small. Try increasing -XX:NewRatio or -XX:NewSize.

Major/Full GC Events: Any Full GC is expensive and should be rare (once per hour or less). If you see multiple Full GCs, your heap size is insufficient for your workload — increase -Xmx.

High Pause Times: Pauses over 500ms on latency-sensitive systems indicate the wrong garbage collector. For sub-10ms pauses, use ZGC or Shenandoah. For 50-200ms, use G1GC. For 500ms+, Parallel GC is acceptable.

FAQ
For Java 11+: java -Xlog:gc*:file=gc.log:time -jar myapp.jar

For Java 8-10: java -XX:+PrintGCDetails -XX:+PrintGCTimeStamps -Xloggc:gc.log -jar myapp.jar

The log file will be created in your current directory. Run your app under realistic load for 5-10 minutes to collect meaningful data.
Pause time is how long the garbage collector stopped your application threads (Stop-The-World pause). During this time, your app doesn't respond to requests or process data. A 100ms pause means your application was frozen for 100 milliseconds. For real-time systems (trading, gaming), keep pauses under 10ms. For web services, 50-200ms is acceptable.
If "Heap After" values increase throughout the log period (e.g., 100MB → 200MB → 400MB), you likely have a memory leak. Objects are being created faster than they can be garbage collected. Use Eclipse MAT, JProfiler, or YourKit to analyze a heap dump and identify which objects are accumulating. Then fix the code that's leaking.
Frequent Full GCs indicate your heap is undersized for your workload. The garbage collector can't keep up and is doing expensive full collections to reclaim space. Solutions: (1) Increase -Xmx (max heap size), (2) Optimize your code to allocate fewer objects, (3) Switch to a different garbage collector like G1GC or ZGC which can handle memory better. Test with a 20-30% larger heap first.
This parser is optimized for Java 11+ -Xlog:gc* format. It will also parse older -XX:+PrintGCDetails format from Java 8-10, though with lower accuracy on some metrics. Format varies significantly between JVM versions. If parsing fails, ensure your log was generated with either flag and paste only the GC lines (not startup messages or application logs).

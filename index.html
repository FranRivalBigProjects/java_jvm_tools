Java memory management guide

Java memory management is handled by the JVM's automatic garbage collector, but tuning it correctly can reduce pause times by orders of magnitude.

# Enable GC logging (Java 11+) — essential for tuning java -Xms4g -Xmx4g -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/var/dumps/ -jar myapp.jar # ZGC for ultra-low latency (Java 15+) java -Xms16g -Xmx16g -XX:+UseZGC -XX:MaxGCPauseMillis=1 -Xlog:gc*:file=gc.log:time -jar myapp.jar

Always set -Xms equal to -Xmx in production
When -Xms (initial heap) is smaller than -Xmx (max heap), the JVM performs expensive heap resize operations as load increases. These resizes trigger full GC pauses. In production, always set -Xms4g -Xmx4g (same value) to give the JVM full memory from startup. The trade-off: your process reserves all memory immediately even when idle. For Kubernetes: set JVM max heap to ~75% of container memory limit to leave room for metaspace, thread stacks, and off-heap buffers.

Reading GC logs for java garbage collection tuning
Enable logs with -Xlog:gc*:file=gc.log:time. Key patterns to look for: frequent minor GC (Eden too small — increase -XX:NewRatio or -XX:NewSize), major/full GC events (heap sizing issue or memory leak), long STW pauses (consider ZGC or Shenandoah), and rapid growth in old generation (memory leak — analyze heap dump with Eclipse MAT). Upload gc.log to GCEasy.io for visual analysis.

Java memory management tools for production
Built-in (free): jstat -gcutil for live GC stats, jmap -heap for heap summary, jcmd for diagnostic commands, VisualVM for GUI monitoring. Third-party: Eclipse MAT for heap dump analysis (identifying memory leaks), JProfiler for CPU + memory profiling, YourKit for production-safe profiling. Cloud APM: Datadog, New Relic, Elastic APM for continuous production GC monitoring with alerting.

Frequently Asked Questions
Top GC tuning tools: GCEasy.io — upload GC log for visual timeline analysis (free tier). VisualVM — free JDK tool, shows live GC activity, heap usage, CPU. jstat — built-in CLI: jstat -gcutil <pid> 1000 prints GC stats every second. JProfiler — commercial, deep GC and allocation profiling. Eclipse MAT — heap dump analysis for memory leak detection. Datadog APM / New Relic — production continuous GC monitoring.
Step-by-step GC tuning: (1) Enable GC logging: -Xlog:gc*:file=gc.log:time. (2) Run under production load and collect logs. (3) Analyze with GCEasy or VisualVM — identify pause patterns. (4) Set -Xms = -Xmx to prevent resize pauses. (5) Choose GC: ZGC for latency, Parallel GC for throughput, G1GC for balance. (6) Set -XX:MaxGCPauseMillis target. (7) Tune generation sizes if using G1GC: -XX:NewRatio. (8) Re-test and iterate.
G1GC (default Java 9+): Concurrent mark-sweep with short STW pauses. Good for heaps 4GB–32GB. Predictable pause targets via -XX:MaxGCPauseMillis. Works everywhere. ZGC (Java 15+ production-ready): Near-zero pause times (<1ms typical) regardless of heap size — handles terabytes. Better for latency-sensitive services. Use ZGC for real-time APIs, trading systems, or any service where GC pause spikes cause SLA violations. Use G1GC for general-purpose backend applications where 200ms pauses are acceptable.
JVM GC (Garbage Collection) automatically reclaims memory occupied by objects no longer referenced. The JVM heap is divided into generations: Young generation (Eden + Survivor spaces) for new short-lived objects — collected frequently with minor GCs. Old generation (tenured) for long-lived objects — collected less frequently with major GCs. Metaspace (Java 8+) for class metadata. GC pauses stop application threads ("Stop-The-World") to maintain heap consistency. Modern GCs (ZGC, Shenandoah) do most work concurrently to minimize these pauses.
Memory leak detection tools: (1) Eclipse MAT — analyzes heap dumps (-XX:+HeapDumpOnOutOfMemoryError), shows dominator tree and leak suspects. (2) VisualVM — heap dump + sampler. (3) JProfiler — allocation tracking, reference graphs. (4) jmap CLI: jmap -dump:format=b,file=heap.hprof <pid>. Signs of a memory leak: old generation grows continuously, full GC runs more frequently but can't reclaim space, OutOfMemoryError: Java heap space after long run.
